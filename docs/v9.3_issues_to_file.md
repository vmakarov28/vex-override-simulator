## Issue 1 — PROBLEM 69: Cup-flip reward exploit (FIXED)

**Title**: `PROBLEM 69: Cup-flip reward exploit lets robots farm cup_orient_correct`

**Labels**: `bug`, `reward-shaping`, `fixed-in-v9.3`

**Body**:
After v9.2b bumped `cup_orient_correct/wrong` to ±0.10, the reward delta
from `wrong → correct` orientation became +0.20 per flip, exceeding
`flip_penalty = -0.15`.  Net: **+0.05 per flip**.  Robots learned to
sit near a goal and alternate cup orientation every step.

### Evidence
- `flip_penalty` in [Rwd] logs consistently −4 to −6 per 82k-step window
  (= ~30–40 cup flips per episode).
- User visually observed "robots repeatedly flipping cups while
  approaching goals" in the final-match video.

### Fix (commit `d94a163`)
Added per-robot `_steps_since_cup_flip` counter.  Suppressed
`cup_orient_correct` bonus for `CUP_FLIP_LOCKOUT_STEPS = 15` after any
`flip_cup` while carrying.  The wrong-orientation **penalty** still
applies during lockout so the robot cannot dodge it by flipping (each
flip restarts the lockout).

## Issue 2 — PROBLEM 70: Pinning resurgence in late training (FIXED)

**Title**: `PROBLEM 70: Pinning re-emerges from 9M→14M steps`

**Labels**: `regression`, `reward-shaping`, `fixed-in-v9.3`

**Body**:
After the v9.2 pinning penalty appeared to be working (pinning dropped
from −559 peak at 2.5M to −90 around 9M), the policy converged onto a
"pin + hold" local optimum in later training:

| Step | pinning |
|------|---------|
| 9.0M | −90  |
| 11.0M | −210 |
| 13.0M | −370 |
| 14.0M | −410 |

Entropy also dropped 5.0 → 4.68 in the same window, consistent with
the policy overfitting to a single strategy.

### Fix (commit `d94a163`)
`pinning_violation` weight bumped −4.0 → −6.5 (1.6×).  Combined with
the existing `PINNING_STEPS_LIMIT=40` threshold (2 s), sustained
pinning now costs ~16% more per violation step.

### Follow-up if it regresses again
- Quadratic penalty growth past the threshold (currently linear)
- Entropy bonus increase in second half of training
- Higher-pressure curriculum (more dense start states)

## Issue 3 — PROBLEM 71: SC5b center-yellow scoring lacks diagnostic (FIXED)

**Title**: `PROBLEM 71: Cannot verify SC5b +10/yellow-half is being applied at match end`

**Labels**: `tooling`, `diagnostics`, `fixed-in-v9.3`

**Body**:
User reported a match where blue scored a yellow half into the center
goal at endgame start; final state had 2 blue / 0 red in midfield
(blue majority); final score showed Red 20 : 16 Blue but expected
Red 20 : 26 Blue (+10 for SC5b yellow claim).

Static analysis of `calculate_final_score` looked correct, but there
was no runtime telemetry to confirm whether the +10 was actually being
applied.

### Fix (commit `d94a163`)
`RulesEngine.calculate_final_score` now prints a one-line `[SC5b]`
diagnostic at every match end with: midfield robot counts per
alliance, computed majority, count of visible yellow halves in the
center goal, and the +pts delta applied to each alliance.

### Next time the bug appears
The diagnostic line will immediately show whether (a) the majority
calc was wrong, (b) the yellow half wasn't visible (likely cause: a
cup got placed dark-side-down on top), or (c) the delta calc was
wrong.  No new instrumentation should be needed.

## Issue 4 — Feature: Rule-based HeuristicBot for play/debug/eval

**Title**: `Feature: HeuristicBot — rule-based perfect-play agent (NEW)`

**Labels**: `enhancement`, `agent`

**Body**:
Added `agents/heuristic_bot.py`, a deterministic priority-tree agent
designed to make "the best move every time" given the live field state.
Reads simulator state every control step and emits actions in either
sim-action dict form or policy `(cont, disc)` array form for drop-in
use with the existing env_wrapper.

### Decision priority (highest first)
1. Emergency intercept of opponent about to score
2. Endgame park (midfield majority + parking bonus)
3. Pre-placement cup flip (orient before arrival)
4. Score into best valid goal (respects stack legality)
5. Pre-placement pin flip (yellow-side UP when alliance owns toggle)
6. Toggle opportunity (flip opp-owned toggles in detour range)
7. Pickup pin (own-color/yellow priority; avoid pure-opp solids)
8. Pickup cup (nearest unclaimed)
9. Default — drive to midfield centre

### Runner: `run_with_bots.py`
```
python run_with_bots.py --bots all
python run_with_bots.py --bots red1,red2 --policy artifacts/models/final.pt
python run_with_bots.py --bots red1 --policy artifacts/models/final.pt
```

### Known limitations / future work
- No multi-step pathing around obstacles
- No alliance-partner coordination (two bots may target same element)
- No multi-step opponent prediction
- Doesn't dive into midfield to break ties at match end

### Use cases
- Sanity-check the simulator (does the game actually work?)
- Debug new reward shaping by watching what the bot does that the
  trained policy doesn't
- Provide a curriculum opponent (training against a heuristic baseline)
- Visual demos that aren't dependent on a converged checkpoint
